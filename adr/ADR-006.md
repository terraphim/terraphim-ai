# ADR 006: Defer Inversion of KG-Router vs Static-Config Precedence

Status: Deferred

Date: 2026-05-22

## Context

The ADF orchestrator (`crates/terraphim_orchestrator`) selects an LLM provider/model for each agent spawn via two layered mechanisms:

1. **Static per-agent config** in `AgentDefinition` (`crates/terraphim_orchestrator/src/config.rs:689`): `cli_tool`, `model`, `provider`, `fallback_provider`, `fallback_model`. These are operator-authored in `conf.d/*.toml`.
2. **KG tier routing** in `spawn_agent` (`crates/terraphim_orchestrator/src/lib.rs:1917-1964`): an Aho-Corasick match over the agent's task prompt against the rules in `docs/taxonomy/routing_scenarios/adf/*.md`. The matched concept's primary route overrides the static `cli_tool` and `model`.

Today the comment at `lib.rs:1919` reads:

> KG routing first (phase-aware tier selection from markdown rules). Takes priority over static model config so tier routing controls selection.

This precedence (KG > static) was originally chosen so that tier-routing rules could drive provider selection centrally as the agent fleet grew: a single edit to `implementation_tier.md` reroutes every implementation-class agent without touching individual agent TOML files. It also makes it possible to centrally migrate agents between providers (e.g. anthropic â†’ kimi) by editing one file.

A failure mode of this precedence was documented in `.docs/research-kg-router-fallback-fix.md` (the same investigation that triggered this ADR): when an operator explicitly chose a non-default `cli_tool` for an agent (e.g. `compliance-watchdog` configured with `cli_tool=opencode, fallback_model=kimi-for-coding/k2p6`), the KG router silently overrides that choice and routes the agent to `anthropic/sonnet` because the task text matches `implementation_tier`. The operator's intent is invisible to the routing layer.

A natural fix is to **invert the precedence**: when both `cli_tool` and `model` are set on an `AgentDefinition`, treat them as authoritative and skip KG routing. KG routing then only kicks in for agents that haven't pinned a CLI/model.

## Decision

**Defer this precedence inversion.** Instead, the immediate KG-router-fallback fix uses a narrowly-scoped opt-in flag (`bypass_kg_routing: bool` on `AgentDefinition`, default false), set automatically by the quota-exit fallback respawn so that the operator-chosen `fallback_provider`/`fallback_model` are honoured verbatim. Healthy-state spawns continue to use KG tier routing as today.

The precedence question (i.e. should `cli_tool` and `model` on `AgentDefinition` *always* dominate KG routing when set?) is recorded here as a deliberate non-decision, to be revisited once the immediate fix has been observed in production and once the next time we have evidence (a new failure or a clearly-blocked operator intent) tips the scales.

## Why Defer

1. **Blast radius**. Inverting precedence changes routing decisions for *every* healthy-state agent that has any `model` field set. The current taxonomy (`docs/taxonomy/routing_scenarios/adf/{implementation,review,planning}_tier.md`) is the operational source of truth for tier-based migrations; flipping the precedence means TOML edits are required to opt into tier routing, not opt out of it.
2. **No present evidence of a second failure mode**. The only documented breakage is the fallback-respawn loop, which the narrowly-scoped `bypass_kg_routing` flag closes. Without a second independent case, an architectural inversion is speculative.
3. **Reversibility cost asymmetry**. If we defer and later need the inversion, we can ship it (an additive default-on logic change with a clear migration path). If we ship the inversion now and need to revert, every agent's TOML may need touching to re-opt into tier routing, with downtime risk on bigbox.
4. **Concept ambiguity**. "When both `cli_tool` and `model` are set" is not a sharp boundary: today's TOMLs set `cli_tool` for almost every agent (it's a required field) but only sometimes set `model`. A precedence rule keyed on "both set" would silently change behaviour for most agents. A clean inversion would also need a TOML migration to add `routing_mode = "auto" | "static" | "kg"` -- which is exactly the kind of speculative flexibility we are eliminating in the design phase of the immediate fix.
5. **Operator workflow is not yet stable**. The ADF taxonomy is still evolving (planning/review/implementation tiers added in 2026-Q1). Pinning a precedence model now risks freezing it before the operational pattern matures.

## Consequences

- **Short term**: the immediate fix (`bypass_kg_routing` flag + `parse_reset_time` extension + 15-min safety floor) closes the demonstrated bug. Operators continue to see KG tier routing dominate static config in healthy state; the only exception is fallback respawns.
- **Continuing risk**: operators who configure `cli_tool=opencode` on an agent will *still* see the KG router push that agent to claude/sonnet in healthy state, despite the explicit config. This is by design under the current precedence and is not a regression. If a user-visible incident occurs because of this gap, that incident becomes the evidence to reopen this ADR.
- **Future revisit triggers**: any one of the following should reopen this decision:
  1. A second independent failure mode caused by KG routing ignoring static config.
  2. Operator feedback that tier routing is more confusing than it is helpful.
  3. A migration where flipping precedence would simplify rather than complicate the TOML migration (e.g. a major taxonomy rewrite).
  4. The arrival of agents whose correct provider is structurally not derivable from prompt text (e.g. a privacy-sensitive agent that must run only on a local model regardless of task content).
- **Documentation**: the deferral is recorded both here and in `.docs/research-kg-router-fallback-fix.md` (the "Multiple Interpretations Considered" and "Eliminated from Scope" tables). Future reviewers can trace the rationale without re-deriving it.

## Alternatives Considered

| Option | Why Rejected For Now |
|--------|----------------------|
| Invert precedence unconditionally (KG only when static is unset) | Behaviour change for every agent; needs broader review and a TOML migration plan |
| Add a `routing_mode` enum (`auto` / `static` / `kg`) on `AgentDefinition` | Speculative three-state flexibility for a two-state problem; the immediate fix only needs an opt-out for fallback respawns |
| Detect operator intent heuristically (`cli_tool != "claude" && model.is_some()` -> static) | Magical and brittle; future agents would silently change behaviour as `cli_tool` defaults drift |
| Remove KG tier routing entirely | Loses the operational benefit (centralised tier migration); not justified by today's evidence |
| Status quo + better docs | Doesn't fix the demonstrated bug -- which is why we're shipping the `bypass_kg_routing` flag in parallel |
