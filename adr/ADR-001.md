# ADR 001: Layered Architecture and Dependency Direction Rules

Status: Accepted

Context
- The orchestrator, tracker, and related crates form a multi-crate workspace with clear dependency direction.
- Current architectural understanding relies on guidelines in architecture-review-report.md and related planning docs, but there is no codified ADR directory capture.

Decision
- Define a layered architecture with explicit dependency rules:
  - Outer layers depend on inner layers, never the reverse.
  - Core domain logic resides in the domain layer; infrastructure adapters depend on domain interfaces.
  - Public crates expose stable APIs; internal crates may depend on abstractions but not on concrete implementations.
- Introduce a standard interface contract pattern for inter-crate communication to prevent leakage of implementation details.

Consequences
- Improves traceability and onboarding by codifying architectural intent.
- Requires updates to CI checks to ensure conformance to the layer boundaries.
