# ADR-007: Decision Tier + pi-rust Routes via Taxonomy Templates

**Status:** Accepted
**Date:** 2026-05-23
**Deciders:** alex
**Related:** ADR-006 (KG-router fallback), #1807, #1816, #1783, #1785

## Context

The ADF KG router scores agent tasks against three taxonomy tiers
(`planning_tier` 80, `implementation_tier` 50, `review_tier` 40). On
2026-05-23 the live fleet shows:

- Anthropic planning route degraded -> planning falls back to implementation
  models (kimi/minimax), weakening multi-file design tasks
- `opencode/gpt-5.5` and `minimax-coding-plan/MiniMax-M2.7-highspeed`
  confirmed live via `opencode models` on bigbox -- both flat-cost
  subscription models
- pi-rust 0.1.10 installed at `/home/alex/.local/bin/pi-rust` with
  provider parity to opencode for `zai-coding-plan`,
  `minimax-coding-plan`, `kimi-for-coding`, and `openai-codex`
- Some classes of work (log triage, nightwatch retrospective,
  post-merge gate, KLS quality evaluation) are analytical decisions
  over execution data and benefit from session continuity -- a fit for
  pi-rust + GPT-5.5

Two parallel issues already track deeper pi-rust integration:

- **#1783** proposes replacing opencode runtime with pi-rust as ADF's
  primary CLI
- **#1785** proposes a native Rust `terraphim_pi_agent` bridge crate
  with provider inference, skill-chain resolution, and swarm orchestrator

## Decision

1. **Add a new KG tier**: `decision_tier` at priority **65**, between
   planning (80) and implementation (50). Synonyms cover log triage,
   retrospective, quality evaluation, post-merge gate, conflict
   resolution.
2. **Expand existing tiers with new model routes**:
   - `planning_tier`: add `opencode/gpt-5.5` and a pi-rust GPT-5.5 route
   - `implementation_tier`: add `MiniMax-M2.7-highspeed` and a pi-rust
     MiniMax route
3. **Invoke pi-rust directly via taxonomy `action::` templates**, not
   via a Rust helper in the orchestrator. Each route carries its own
   argv template; the KG router already renders it.
4. **Do not build `terraphim_pi_agent`** as part of this change. When
   #1785 lands, the bridge crate supersedes pi-rust direct invocations
   cleanly (only the `action::` lines change).
5. **Reuse `bypass_kg_routing` with `fallback_provider="/home/alex/.local/bin/pi-rust"`**:
   the existing `fallback_provider` field is a filesystem path. Setting
   it to the pi-rust path makes the orchestrator's existing respawn
   logic substitute the binary; no new flag needed.

## Consequences

### Positive
- No Rust code change required for the integration; pure markdown + ADR
- Decision tier separates analytical work from implementation, removing
  routing ambiguity for log-analyst / nightwatch / retrospective agents
- pi-rust provides a *drop-in alternative spawner* for the same
  subscription providers: a failing opencode binary triggers a re-spawn
  through pi-rust against the same provider+model, not a tier downgrade
- `opencode/gpt-5.5` adds healthy planning capacity while Anthropic is
  degraded
- `MiniMax-M2.7-highspeed` adds higher-throughput implementation
  capacity for fast-turn coding agents
- Coordinates cleanly with #1783 and #1785; does not duplicate either

### Negative
- Two CLIs invoked for the same model (opencode and pi-rust) double
  the auth surface and quota tracking until #1785's bridge crate
  consolidates them
- `action::` template duplication: each tier needs its own pi-rust line
  (mitigated: only 3 tiers x 2 routes = small)

### Rejected alternatives
- **Per-agent opt-in flag for pi-rust**: rejected as config drift; new
  tier bucket is cleaner
- **Add `terraphim_pi_agent` bridge crate now**: rejected to avoid
  duplicating #1785, which has a fuller scope (provider inference,
  skill chains, swarm orchestrator)
- **Tighten Anthropic timeout / disable Anthropic outright**: rejected
  as out of scope; the issue is operational, not architectural
