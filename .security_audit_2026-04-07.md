## Critical Findings (MUST FIX)

### 1. CVE: rustls-webpki 0.102.8 - Invalid CRL Handling
**Severity**: CRITICAL
**ID**: RUSTSEC-2026-0049
**Date**: 2026-03-20
**Impact**: CRLs (Certificate Revocation Lists) are not properly validated due to faulty matching logic in Distribution Point processing

**Affected Component Chain**:
```
rustls-webpki 0.102.8 → rustls 0.22.4 → tungstenite 0.21.0 →
tokio-tungstenite 0.21.0 → serenity 0.12.5 → terraphim_tinyclaw 1.16.9
```

Also affects: `tokio-rustls`, `terraphim_mcp_server`, multiple agent systems

**Evidence**:
```bash
Crate:     rustls-webpki
Version:   0.102.8
Title:     CRLs not considered authoritative by Distribution Point due to faulty matching logic
Date:      2026-03-20
ID:        RUSTSEC-2026-0049
URL:       https://rustsec.org/advisories/RUSTSEC-2026-0049
Solution:  Upgrade to >=0.103.10
```

**Remediation**: Upgrade `rustls-webpki` to version 0.103.10 or later. This requires coordinating upstream dependency updates through `rustls`, `tokio-rustls`, and `tungstenite`.

**Command**:
```bash
cargo update rustls-webpki --aggressive
cargo audit  # verify resolution
```

### 3. Network Exposure Risk

**Severity**: HIGH
**Finding**: Two services listening on all interfaces (0.0.0.0)

```
LISTEN 0 1024 0.0.0.0:3456 - terraphim-llm-p (LLM process)
LISTEN 0 4096 0.0.0.0:11434 - Ollama API endpoint
LISTEN 0 4096 *:80  - HTTP (expected)
LISTEN 0 4096 *:443 - HTTPS (expected)
```

**Risk**: Unauthenticated access to:
- LLM inference engine (port 3456)
- Ollama API (port 11434) - allows model queries and management

**Remediation**:
1. Bind services to loopback (127.0.0.1:3456) unless multi-host deployment required
2. If multi-host: implement authentication (API tokens, mTLS)
3. Use firewall rules to restrict network access
4. Deploy behind reverse proxy with authentication

**Configuration**:
```bash
# Preferred: localhost only
OLLAMA_HOST=127.0.0.1:11434
TERRAPHIM_LLM_HOST=127.0.0.1:3456
```
