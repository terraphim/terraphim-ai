Title: Add CI pre-flight step: assert no unexpected 0.0.0.0 port bindings
WIG: WIG-4 (Production-ready build and release pipeline — quality gates enforced)
Compound-RICE estimate: reach=25 impact=7 confidence=0.9 synergy=1.4 effort=1 maintenance=1.1 -> score=200.5
Problem: Two P1 security findings (Ollama :11434, dlt-postgres :15432) persisted across multiple sentinel cycles because there was no automated gate to detect them. The current CI pipeline contains no step that asserts which ports are bound to 0.0.0.0, so regressions are invisible until the next manual audit. Without this gate, #1374 remediation can silently regress after merge.
Proposed solution:
- Add `scripts/ci/check-port-exposure.sh`: runs `ss -tlnp` (Linux CI) and exits non-zero if any address matches `0.0.0.0` outside an explicit allowlist (e.g. `127.0.0.1:*` loopback entries are permitted)
- Allowlist stored as a newline-delimited flat file `scripts/ci/allowed-bindings.txt`; adding a legitimate public port requires a deliberate PR change to that file
- Wire as a mandatory job step in `.github/workflows/ci.yml` before integration tests run, so the job fails fast if unexpected exposure is present
- Script emits structured output: `EXPOSED: 0.0.0.0:PORT (process)` lines for each violation, making the failure self-documenting in CI logs
Acceptance criteria (mini-UAT):
  Given a CI runner where no additional services are started
  When the `check-port-exposure.sh` step runs
  Then it exits 0 and prints "No unexpected 0.0.0.0 bindings found"
  And if a test helper binds to 0.0.0.0:15432 (simulating the dlt-postgres regression)
  And the port is not in `allowed-bindings.txt`
  Then the step exits non-zero with output "EXPOSED: 0.0.0.0:15432" and the CI job fails before integration tests execute
Marketing hint: "terraphim-ai CI now fails fast on any accidental public port exposure — security regressions caught at merge, not at the next audit."
Suggested agent: @adf:devops

---NEW_ISSUE---
Title: Expose /health endpoint on MCP server and add CI readiness wait loop
WIG: WIG-3 (Complete agent orchestration and MCP integration) and WIG-4 (CI quality gates)
Compound-RICE estimate: reach=15 impact=7 confidence=0.9 synergy=1.6 effort=2 maintenance=1.0 -> score=75.6
Problem: MCP integration tests (extended by #1354) can start before the MCP server process has finished binding its port, producing non-deterministic failures that look like protocol errors but are actually race conditions. There is currently no way for CI or a local developer to probe server readiness without parsing log output. This contributes to the test-infrastructure fragility tracked in #1375 and undermines confidence in WIG-3 sign-off.
Proposed solution:
- Add a `GET /health` route to `crates/terraphim_mcp_server/src/main.rs` returning `{"status":"ok","transport":"sse|stdio","uptime_secs":N}` with HTTP 200; no authentication required
- Return HTTP 503 with `{"status":"starting"}` during the brief window before the automata are loaded, ensuring the readiness signal is accurate
- Add `scripts/ci/wait-for-mcp.sh MAX_SECONDS URL`: polls `/health` with `curl --silent --fail` every 500 ms and exits 0 on first 200 response; exits non-zero and prints elapsed time if MAX_SECONDS is exceeded
- Wire the wait script before the MCP integration test step in `.github/workflows/ci.yml`
Acceptance criteria (mini-UAT):
  Given the MCP server is started with `cargo run -p terraphim_mcp_server`
  When `curl -s http://localhost:PORT/health` is executed after the server is ready
  Then the response body contains `"status":"ok"` and HTTP 200 is returned
  And when `wait-for-mcp.sh 30 http://localhost:PORT/health` is called immediately after process launch
  Then it blocks until the server is ready and exits 0 within 30 seconds
  And if the server never starts, the script exits non-zero after the timeout with a clear error message
Marketing hint: "MCP server now signals readiness over HTTP — eliminates an entire class of CI race failures and makes local dev loops deterministic."
Suggested agent: @adf:backend
