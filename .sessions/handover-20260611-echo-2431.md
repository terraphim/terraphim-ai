## What Was Done

### Problem
`FirecrackerExecutor::list_snapshots()` in `crates/terraphim_rlm/src/executor/firecracker.rs:646-668`
always returned `Vec::new()` with a TODO comment claiming `SnapshotManager.list_snapshots`
requires `&mut self` access that was unavailable.

### Fix
Applied the identical `tokio::sync::Mutex` pattern already used in `delete_session_snapshots()`
on the same struct. Three logical changes:

1. `list_snapshots()` now acquires `self.snapshot_manager.lock().await` and calls
   `snapshot_manager.list_snapshots(Some(&vm_id)).await.unwrap_or_default()`
2. Maps each returned fcctl-core snapshot (`snapshot.id`) to a `SnapshotId::new(&snapshot.id, *session_id)`
3. Returns empty list (not error) when no VM assigned or SnapshotManager uninitialised

### Tests Added
Two unit tests in the `#[cfg(test)]` block inside `firecracker.rs`:
- `test_list_snapshots_no_vm_assigned` — no VM → empty list
- `test_list_snapshots_snapshot_manager_uninitialised` — VM assigned but manager not initialised → empty list

Both guard on `is_kvm_available()` (standard pattern for all FirecrackerExecutor tests).
